# DECISIONS

## 2026-06-19 Harness-based Development 문서 체계 도입

### 결정

프로젝트 루트에 `AGENTS.md`, `PROJECT_STATE.md`, `TASKS.md`, `DECISIONS.md`, `EVIDENCE.md`, `CHANGELOG.md`를 운영 문서 세트로 유지한다.

### 이유

- 기능 구현보다 반복 가능한 개발 루프와 검증 증거를 우선하기 위해서다.
- Codex가 매 작업 시작 시 현재 상태, 결정, 검증 증거를 확인할 수 있게 하기 위해서다.
- 단발성 코드 생성이 아니라 장기 유지보수 가능한 프로젝트로 운영하기 위해서다.

### 영향

- 신규 작업은 Clarify → Plan → Execute → Verify → Evidence → Decide 순서를 따른다.
- 아키텍처 변경, 라이브러리 교체, 대규모 리팩토링, 데이터 구조 변경, 기존 기능 제거는 사용자 승인 후 진행한다.
- 완료 선언 전 검증 명령과 증거 기록을 남긴다.

## 2026-06-19 문서 전용 변경의 검증 범위

### 결정

이번 작업은 앱 코드 변경이 아니므로 전체 `npm run quality` 대신 문서 존재 확인, Git 상태 확인, 가능한 경량 검증을 우선한다.

### 이유

- 문서 생성 작업에서 전체 Playwright/빌드 검증은 비용이 높다.
- 단, 운영 원칙상 증거가 필요하므로 명령 실행 결과는 `EVIDENCE.md`에 남긴다.

### 영향

- 기능 코드 변경 작업에서는 기존 원칙대로 build/test/lint 우선순위를 적용한다.
- 문서 전용 작업에서는 문서 존재성 및 Git 상태를 필수 증거로 남긴다.
