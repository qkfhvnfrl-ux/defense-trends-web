# DECISIONS

## 2026-06-19 Harness-based Development 문서 체계 도입

### 결정

프로젝트 루트에 `AGENTS.md`, `PROJECT_STATE.md`, `TASKS.md`, `DECISIONS.md`, `EVIDENCE.md`, `CHANGELOG.md`를 운영 문서 세트로 유지한다.

### 이유

- 기능 구현보다 반복 가능한 개발 루프와 검증 증거를 우선하기 위해서다.
- Codex가 매 작업 시작 시 현재 상태, 결정, 검증 증거를 확인할 수 있게 하기 위해서다.
- 단발성 코드 생성이 아니라 장기 유지보수 가능한 프로젝트로 운영하기 위해서다.

### 영향

- 신규 작업은 Clarify → Plan → Execute → Verify → Evidence → Decide 순서를 따른다.
- 아키텍처 변경, 라이브러리 교체, 대규모 리팩토링, 데이터 구조 변경, 기존 기능 제거는 사용자 승인 후 진행한다.
- 완료 선언 전 검증 명령과 증거 기록을 남긴다.

## 2026-06-19 문서 전용 변경의 검증 범위

### 결정

이번 작업은 앱 코드 변경이 아니므로 전체 `npm run quality` 대신 문서 존재 확인, Git 상태 확인, 가능한 경량 검증을 우선한다.

### 이유

- 문서 생성 작업에서 전체 Playwright/빌드 검증은 비용이 높다.
- 단, 운영 원칙상 증거가 필요하므로 명령 실행 결과는 `EVIDENCE.md`에 남긴다.

### 영향

- 기능 코드 변경 작업에서는 기존 원칙대로 build/test/lint 우선순위를 적용한다.
- 문서 전용 작업에서는 문서 존재성 및 Git 상태를 필수 증거로 남긴다.
# 2026-06-19 장비 검색 카탈로그를 제품 중심축으로 채택

## 결정

웹페이지의 1차 사용 목적을 회의 브리핑이 아니라 `전 세계 지상 장비 검색 카탈로그`로 둔다.

## 이유

- 사용자가 팀원에게 필요한 핵심 목적을 장비 검색이라고 명확히 했다.
- 장비 데이터는 계속 확장될 예정이므로, 첫 화면에서 검색/분류/상세 진입이 바로 가능해야 한다.
- 기존 브리핑형 대시보드, 비교 페이지, 기술/사례 분리 메뉴는 검색 목적에서는 초점을 분산시킨다.

## 영향

- 상단 메뉴를 `장비 검색 / 전체 장비 / 전장 인사이트 / 출처`로 축소했다.
- `Compare`는 별도 메뉴에서 제외하고, 상세 화면의 유사 장비 추천으로 대체했다.
- `Technologies`, `Cases`, `Development`는 `전장 인사이트` 화면으로 통합했다.
- 기존 데이터는 삭제하지 않고 검색/상세/인사이트/출처 화면에서 재사용한다.

# 2026-06-19 3D 기능 단순화

## 결정

현재 버전에서 3D 캔버스를 기본 표시하지 않고, GLB 추후 연동 슬롯과 부품 스펙 목록으로 단순화한다.

## 이유

- 실제 GLB 모델이 아직 준비되지 않았고, 사용자는 3D를 추후 추가하겠다고 밝혔다.
- 플레이스홀더 3D는 팀원의 장비 검색 목적에서는 우선순위가 낮다.
- 3D 런타임을 기본 렌더링에서 제외하면 번들 크기와 화면 복잡도가 줄어든다.

## 영향

- `ModelViewer`는 캔버스 대신 준비 슬롯과 부품 스펙 버튼을 렌더링한다.
- Playwright 검증은 캔버스 대신 `.model-slot`과 `.component-slot`을 확인한다.
