# 2026-06-20 단일 장비 공유는 선택 패널에서 처리

## 결정

선택한 장비 하나의 요약 공유는 검색 결과 전체 내보내기에 섞지 않고, 선택 장비 패널의 `요약 복사` 작업으로 제공한다.

## 이유

- 팀 대화에서는 전체 검색 결과보다 특정 장비 하나의 핵심 정보만 전달하는 경우가 많다.
- 검색 결과 CSV/요약은 목록 공유용이고, 선택 패널은 현재 보고 있는 장비의 판단 근거를 전달하는 위치다.
- 새 데이터 구조 없이 기존 장비, 계열, 전장 사례, 기술, 출처 상태를 조합해 생성할 수 있다.

## 영향

- 선택 장비 패널 작업은 `요약 복사`, `상세 페이지 열기` 2개로 유지한다.
- Clipboard API 실패 시 상태 메시지로 알려준다.
- 전체 결과 CSV/요약 동작은 변경하지 않는다.

# 2026-06-20 전체 장비 메뉴는 숨기고 라우트 호환은 유지

## 결정

상단 메뉴에서 `전체 장비`를 제거하고, 팀원이 보는 기본 메뉴는 `장비 검색`, `전장 인사이트`, `출처` 3개로 유지한다.

## 이유

- 현재 `/equipment`는 `장비 검색`과 같은 카탈로그 화면을 열어 팀원에게 중복 선택지로 보인다.
- 목표는 기능을 많이 보이는 것이 아니라 필요한 기능만 빠르게 보이게 하는 것이다.
- 기존 공유 링크와 상세 페이지 진입은 깨뜨리면 안 되므로 라우트 자체는 유지한다.

## 영향

- `/equipment`와 `/equipment/:id`는 계속 렌더링된다.
- 해당 경로에서도 `장비 검색` 메뉴가 활성화된다.
- 검증 스크립트는 상단 메뉴가 3개인지와 `전체 장비` 버튼이 없는지 확인한다.

## 결정

첫 화면 작업 큐는 새 워크플로 화면을 만들지 않고, 기존 장비 검색 프리셋과 출처 인덱스 필터로 연결한다.

## 이유

- 현재 제품 목표는 팀원이 필요한 기능만 빠르게 쓰는 검색용 웹페이지다.
- 별도 업무 관리 화면을 추가하면 구조가 커지고, 현재 데이터 품질 점검에는 기존 검색/출처 화면이 충분하다.
- 큐는 “어디부터 볼지”를 알려주는 진입점이어야 하며, 실제 작업은 이미 검증된 필터/출처 화면에서 이어지는 편이 안정적이다.

## 영향

- 작업 큐 수치는 현재 정적 데이터에서 계산된다.
- `출처 재확인`은 `/sources?freshness=stale` query를 사용한다.
- 현재 재확인 필요 출처가 0건이면 빈 상태 안내를 정상 결과로 처리한다.

# 2026-06-19 출처 내보내기는 현재 필터 결과 기준으로 생성

## 결정

`/sources`의 요약 복사와 CSV 다운로드는 전체 출처 목록이 아니라 현재 검색어, 출처 유형, 확인 상태 필터가 적용된 결과만 대상으로 한다.

## 이유

- 전체 출처 덤프보다 현재 필터 결과가 회의/메신저/표 작업에서 바로 사용할 수 있다.
- 서버 저장소 없이 정적 Pages에서 동작해야 하므로 클라이언트에서 즉시 생성한다.

## 영향

- CSV 파일명은 반복 작업 편의를 위해 `source-index-results.csv`로 고정한다.
- 출처 URL의 실시간 접속 상태는 포함하지 않고, 현재 데이터의 `checkedAt` 기반 확인 상태만 포함한다.

# 2026-06-19 출처 확인 상태는 정적 checkedAt 기준으로 계산

## 결정

`/sources` 화면의 출처 확인 상태는 각 출처의 `checkedAt` 날짜를 기준으로 180일 이내면 `최근 확인`, 180일 초과면 `재확인 필요`로 계산한다.

## 이유

- GitHub Pages 정적 배포 구조에서는 사용자가 화면을 열 때마다 모든 외부 URL을 실시간 점검하는 방식이 적합하지 않다.
- 실시간 URL 점검은 네트워크 오류, 403, 429, 사이트별 차단 정책에 영향을 받으므로 UI의 기본 상태로 삼기 어렵다.
- `checkedAt` 기반 상태는 재현 가능하고, 데이터 관리자가 어떤 출처부터 다시 확인해야 하는지 빠르게 판단할 수 있다.

## 영향

- `/sources`에서 검색어, 출처 유형, 확인 상태로 출처를 좁힐 수 있다.
- 실제 접속 가능 여부 확인은 기존 `npm run sources:check`를 별도 검증 작업으로 유지한다.
- 향후 `sources:check` 결과를 JSON으로 저장하면 같은 UI에 링크 상태 필터를 추가할 수 있다.

# 2026-06-19 검색 조건은 URL query string으로 공유

## 결정

장비 검색 조건은 서버 저장이나 계정 기능 없이 URL query string으로 공유한다.

## 이유

- 현재 앱은 정적 GitHub Pages 배포 구조라 서버 저장소나 로그인 기능이 없다.
- 팀원이 메신저, 문서, 회의 자료에서 같은 검색 결과를 열려면 URL 기반 복원이 가장 단순하고 안정적이다.
- 필터 상태가 브라우저 URL에 남으면 새로고침, 뒤로가기, 링크 전달 동작을 자연스럽게 지원할 수 있다.

## 영향

- 검색 화면은 `q`, `family`, `category`, `role`, `country`, `status`, `maturity`, `eq` 파라미터를 읽고 쓴다.
- 상세/출처/인사이트 화면은 기존 라우팅을 유지하고, 검색 화면에서만 query string 동기화를 수행한다.
- Playwright 검증은 필터 적용 후 URL 파라미터 생성과 reload 후 결과 복원을 확인한다.

- 사용자가 팀원에게 필요한 핵심 목적을 장비 검색이라고 명확히 했다.
- 장비 데이터는 계속 확장될 예정이므로, 첫 화면에서 검색/분류/상세 진입이 바로 가능해야 한다.
- 기존 브리핑형 대시보드, 비교 페이지, 기술/사례 분리 메뉴는 검색 목적에서는 초점을 분산시킨다.

## 영향

- 상단 메뉴를 `장비 검색 / 전체 장비 / 전장 인사이트 / 출처`로 축소했다.
- `Compare`는 별도 메뉴에서 제외하고, 상세 화면의 유사 장비 추천으로 대체했다.
- `Technologies`, `Cases`, `Development`는 `전장 인사이트` 화면으로 통합했다.
- 기존 데이터는 삭제하지 않고 검색/상세/인사이트/출처 화면에서 재사용한다.

# 2026-06-19 3D 기능 단순화

## 결정

현재 버전에서 3D 캔버스를 기본 표시하지 않고, GLB 추후 연동 슬롯과 부품 스펙 목록으로 단순화한다.

## 이유

- 실제 GLB 모델이 아직 준비되지 않았고, 사용자는 3D를 추후 추가하겠다고 밝혔다.

- 3D 런타임을 기본 렌더링에서 제외하면 번들 크기와 화면 복잡도가 줄어든다.

## 영향

- `ModelViewer`는 캔버스 대신 준비 슬롯과 부품 스펙 버튼을 렌더링한다.
- Playwright 검증은 캔버스 대신 `.model-slot`과 `.component-slot`을 확인한다.

# 2026-06-19 3D 준비 영역을 공개 장치 스펙 패널로 전환

## 결정

현재 버전에서 3D/GLB 준비 영역을 화면에 표시하지 않고, 장비별 공개 장치/부품 스펙 패널을 기본 상세 보조 패널로 사용한다.

## 이유

- 사용자의 현재 목적은 장비 검색용 카탈로그이며, 3D는 추후 추가 대상이다.
- 실제 GLB 모델이 없는 상태에서 3D 준비 문구를 계속 노출하면 팀원이 현재 사용 가능한 기능으로 오해할 수 있다.
- 이미 수집된 `components.json` 데이터는 장비 검색, 스펙 확인, 출처 검토에 바로 가치가 있다.

## 영향

- UI는 장비 검색과 공개 스펙 확인에 더 집중된다.
- 3D 재도입 시에는 별도 결정으로 `modelPath`, `hotspotPosition`, 3D 의존성을 다시 연결한다.
- 검증 스크립트는 3D 슬롯 존재가 아니라 장치 스펙 패널 존재를 기준으로 한다.

# 2026-06-19 검색 결과 단계에서 출처 신뢰도를 노출

## 결정

장비 검색 결과 행과 선택 장비 요약 패널에 출처 신뢰도, 최근 확인일, 공개 출처 수를 직접 표시한다.

## 이유

- 팀원이 장비를 검색한 뒤 바로 회의나 보고서에 활용하려면 데이터의 근거 수준을 상세 페이지로 들어가기 전에 판단할 수 있어야 한다.
- 이미 `sourceConfidenceScore`, `sources.checkedAt`, `sources.length` 데이터가 있으므로 스키마 변경 없이 구현할 수 있다.
- CSV와 요약 복사에도 같은 정보를 포함하면 공유된 검색 결과만 보아도 검증 맥락이 유지된다.

## 영향

- 검색 행의 정보량은 늘어나지만, 배지와 보조 라인으로 분리해 모바일 overflow 없이 표시한다.
- 출처 신뢰도는 자동 검증 결과가 아니라 현재 데이터셋의 공개 출처 기반 참고 지표임을 유지한다.
- 향후 출처 정기 재검증을 자동화하면 같은 표시 영역에 최신 검증일을 계속 반영할 수 있다.

# 2026-06-19 출처 신뢰도를 검색 필터로 승격

## 결정

검색 결과에 표시되던 출처 신뢰도 등급을 별도 필터로 제공하고, URL 공유 상태에도 포함한다.

## 이유

- 팀원이 모든 장비를 훑지 않고 검증이 필요한 `Low` 신뢰도 장비만 빠르게 좁힐 수 있어야 한다.
- 이미 `sourceConfidenceScore`와 `confidenceLabel` 기준이 있으므로 데이터 스키마를 늘리지 않고 구현할 수 있다.
- 검색 조건 공유 링크에 포함하면 검증 요청이나 데이터 보강 요청을 팀원에게 바로 전달할 수 있다.

## 영향

- 검색 필터 수는 4개에서 5개로 늘어난다.
- `confidence=High|Medium|Low` query string이 검색 화면에서 읽고 쓰인다.
- 향후 자동 URL 재검증 점수가 도입되면 같은 필터 기준을 유지하면서 데이터만 갱신할 수 있다.

# 2026-06-19 전장 사례 유무를 검색 필터로 제공

## 결정

장비 검색에 전장 사례 유무 필터를 추가하고, 검색 공유 URL에 `cases` 파라미터로 포함한다.

## 이유

- 팀원은 실제 운용 사례가 확인된 장비를 먼저 검토해야 하는 경우가 많다.
- 사례가 없는 장비도 데이터 보강 대상으로 식별할 수 있어 장비 DB 확장 작업에 도움이 된다.
- 기존 `battlefieldCaseIds` 필드만으로 구현할 수 있어 데이터 스키마 변경이 필요 없다.

## 영향

- 검색 필터 수는 5개에서 6개로 늘어난다.
- `cases=with-cases|without-cases` query string이 검색 화면에서 읽고 쓰인다.
- 결과 요약과 CSV에는 전장 사례 수가 포함된다.

# 2026-06-19 검색 결과 정렬 기준 제공

## 결정

장비 검색 결과에 정렬 기준을 추가하고, 검색 공유 URL에 `sort` 파라미터로 포함한다.

## 이유

- 팀원은 필터링 이후에도 어떤 장비를 먼저 볼지 빠르게 정해야 한다.
- 출처 신뢰도, 전장 사례 수, 계열 수, 최근 확인일은 현재 데이터에서 바로 계산 가능한 실무 우선순위 기준이다.
- 정렬 상태를 URL에 포함하면 같은 우선순위 관점을 팀원에게 그대로 전달할 수 있다.

## 영향

- 검색 필터/정렬 select 수는 6개에서 7개로 늘어난다.
- `sort=confidence-desc|cases-desc|variants-desc|updated-desc` query string이 검색 화면에서 읽고 쓰인다.
- 결과 요약과 CSV는 화면에 표시된 정렬 순서를 그대로 따른다.
