# 프로젝트 운영 원칙

이 프로젝트는 단순 코드 생성 프로젝트가 아니다.

목표는 기능 구현이 아니라 유지보수 가능하고 검증 가능하며 반복 가능한 개발 시스템을 구축하는 것이다.

모든 작업은 다음 루프를 따라야 한다.

입력(Input)
→ 요구사항 명확화(Clarify)
→ 계획(Plan)
→ 구현(Execute)
→ 검증(Verify)
→ 증거(Evidence)
→ 의사결정(Decide)

코드를 작성하는 것보다 위 과정을 유지하는 것을 우선한다.

---

# 기본 원칙

1. 기존 코드베이스를 존중한다.
2. 현재 구조를 먼저 이해한다.
3. 임의로 재설계하지 않는다.
4. 기존 기능을 깨뜨리지 않는다.
5. 검증되지 않은 완료를 선언하지 않는다.
6. 결과보다 증거를 우선한다.
7. 프로젝트 상태를 지속적으로 유지한다.

---

# 작업 시작 규칙

새로운 요청을 받으면 즉시 코드를 수정하지 않는다.

반드시 아래 순서로 진행한다.

## STEP 1. 현재 상태 파악

다음을 먼저 확인한다.

* PROJECT_STATE.md
* TASKS.md
* DECISIONS.md
* EVIDENCE.md
* CHANGELOG.md

그리고 현재 프로젝트 상태를 요약한다.

---

## STEP 2. 요구사항 명확화

다음을 정리한다.

* 사용자가 원하는 최종 결과
* 현재 구현 상태
* 부족한 정보
* 필요한 가정

요구사항이 모호하면 질문을 먼저 수행한다.

---

## STEP 3. 계획 수립

구현 전에 반드시 계획을 작성한다.

계획에는 다음이 포함되어야 한다.

### 목표

무엇을 구현하는가

### 수정 파일

어떤 파일을 수정하는가

### 영향 범위

어떤 기능에 영향을 주는가

### 리스크

발생 가능한 문제

### 검증 방법

완료 여부를 어떻게 확인하는가

사용자 승인 전까지 구현하지 않는다.

---

# 구현 규칙

## 구현 범위 준수

승인된 범위만 수정한다.

임의 기능 추가 금지.

---

## 기존 기능 보호

기존 기능을 삭제하지 않는다.

기존 기능을 단순화하지 않는다.

기존 기능을 데모 수준으로 축소하지 않는다.

---

## 구조 유지

컴포넌트 구조 유지.

데이터 구조 유지.

불필요한 리팩토링 금지.

---

## 확장성 고려

하드코딩 최소화.

재사용 가능한 구조 우선.

향후 기능 추가를 고려한다.

# 상태 머신

모든 작업은 다음 상태 중 하나를 가진다.

new_task

interviewing

planned

executing

verifying

completed

needs_human

failed_with_evidence

retry_with_revision

상태 전이는 명시적으로 기록한다.

---

# 검증 규칙

구현 후 반드시 검증한다.

우선순위는 다음과 같다.

1. Build
2. Test
3. Lint
4. Manual Verification

예시

npm run build

npm run test

npm run lint

검증 실패 시 완료 처리 금지.

---

# 증거 규칙

모든 완료 작업은 증거를 남긴다.

증거에는 다음이 포함되어야 한다.

* 실행 명령어
* 테스트 결과
* 오류 로그
* 수정 내역
* 빌드 결과
* 남은 리스크

증거가 없으면 완료가 아니다.

# 프로젝트 철학

좋은 AI보다 좋은 개발 루프가 중요하다.

좋은 프롬프트보다 좋은 상태 관리가 중요하다.

좋은 결과물보다 재현 가능한 개발 프로세스가 중요하다.

프로젝트는 항상 다음 순서를 따른다.

Input
→ Clarify
→ Plan
→ Execute
→ Verify
→ Evidence
→ Decide

이 원칙을 모든 작업의 최우선 기준으로 삼는다.
